Scripting guide

Pre-request and test scripts run as plain JavaScript against a Postman-compatible pm.* API, so most existing Postman scripts work unmodified. Assertions use Chai's BDD style via pm.expect.

Where scripts live

  • Per request — the Scripts tab has two boxes: Pre-request script (runs before the request is sent) and Test script (runs after the response arrives). Stored in frontmatter as scripts.pre-request / scripts.test.
  • Per folder or collection — a folder.json or collection.json sidecar can reference a script file (e.g. "preRequestScript": "./collection.pre.js") that wraps every request underneath it. Folder scripts wrap direct children; collection scripts wrap every request in the collection.
# ---
# scripts:
#   pre-request: |
#     pm.variables.set("ts", Date.now());
#   test: |
#     pm.test("status is 200", () => pm.response.to.have.status(200));
#     pm.test("has email", () => {
#       pm.expect(pm.response.json().email).to.be.a("string");
#     });
# ---

The pm.* API

pm.request

Available in both phases. Read and adjust the outgoing request before it's sent.

MemberSignature
pm.request.methodstring
pm.request.urlstring
pm.request.headers.get(name)→ string | undefined (case-insensitive)
pm.request.headers.add({ key, value })→ void
pm.request.headers.toObject()→ Record<string,string>

pm.response

Test scripts only. Reading it from a pre-request script throws — there's no response yet.

MemberSignature
pm.response.codenumber — HTTP status code
pm.response.statusstring — status reason phrase
pm.response.responseTimenumber — milliseconds
pm.response.headers.get(name)→ string | undefined
pm.response.text()→ string — raw body
pm.response.json()→ unknown — throws a clear error if the body isn't valid JSON
pm.response.to.have.status(200 | "OK")assert on status code or reason phrase
pm.response.to.have.header(name)assert a header is present
pm.response.to.have.jsonBody()assert the body parses as JSON

Variable scopes

pm.variables, pm.environment, pm.globals, and pm.collectionVariables are all the same underlying store — a deliberate Postman-compat shim. Whichever one an imported script calls, .set() writes to the session at runtime, and .get() reads session first, falling back to the environment.

MemberSignature
pm.variables.get(name)→ string | undefined
pm.variables.set(name, value)→ void — non-string values are JSON-stringified
This is a snapshot, not a live edit
A script's reads see the session as it was when the script started, plus its own writes so far. Its writes are collected and applied once the script finishes — a script never mutates the session out from under itself mid-run.

pm.test & pm.expect

MemberSignature
pm.test(name, fn)Registers a named test. fn may be sync (throw to fail) or return a Promise (rejection fails the test); omit fn to record a bare pass.
pm.expectChai's expect, for BDD-style assertions: pm.expect(value).to.equal(x), .to.be.a("string"), etc.
 
pm.test("has email", () => {
  const body = pm.response.json();
  pm.expect(body.email).to.be.a("string");
  pm.expect(body.role).to.equal("admin");
});

pm.info

{ requestName, iteration } — the current request's name, and the current data-driven iteration index (see workflows).

pm.sendRequest

Fire an extra request from inside a script — useful for fetching a token before the main request runs. Accepts a URL string or a Postman-style request object, and supports both callback and await styles: 

// promise style
const res = await pm.sendRequest({
  url: "https://auth.example.com/token",
  method: "POST",
  header: { "Content-Type": "application/json" },
  body: '{"grant_type":"client_credentials"}'
});
pm.variables.set("TOKEN", res.json().access_token);

// callback style
pm.sendRequest("https://example.com/ping", (err, res) => {
  if (!err) console.log(res.code);
});

The response passed back has the same shape as pm.response above (code, status, responseTime, headers.get(), text(), json()).

console

console.log/info/warn/error/debug are captured (not printed to a real terminal) and shown alongside the test results after the script runs.

Timeouts

Each script phase has a whole-script budget — 10 seconds by default — covering both synchronous execution and any pending promises (chained pm.sendRequest calls, async pm.test bodies). A script that runs past its budget is stopped and reported as timed out.

Full example

#!/usr/bin/env bash
# ---
# description: Fetches a single user record by id
# scripts:
#   pre-request: |-
#     pm.variables.set("RUN_ID", String(Date.now()));
#   test: |-
#     pm.test("status is 200", () => pm.response.to.have.status(200));
#     pm.test("echoes the run id", () => {
#       const body = pm.response.json();
#       pm.expect(body.headers["X-Run-Id"]).to.equal(pm.variables.get("RUN_ID"));
#     });
# ---

BASE_URL="${BASE_URL:-httpbin.org}"
RUN_ID="${RUN_ID:-local}"

curl --request GET \
  --url "https://${BASE_URL}/get" \
  --header "X-Run-Id: ${RUN_ID}"
The sandbox has no network access of its own
Scripts can't open sockets directly — pm.sendRequest is the only way out, and it's routed through the same request engine as every other send. This keeps Freepost's "only one module ever opens a socket" guarantee intact even with untrusted or imported scripts.

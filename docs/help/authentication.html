Authentication

Freepost handles the common auth schemes two ways: simple ones (Basic, Bearer, API key) are just headers plus variables, and OAuth2 is a structured helper on the request's Auth tab that fetches a token for you and drops it into a session variable.

Basic, Bearer & API key

These need no special machinery — they're ordinary request fields, so they stay valid, runnable curl on disk:

  • Bearer / API key — add an Authorization (or custom) header whose value references a variable, e.g. Authorization: Bearer ${TOKEN} or X-API-Key: ${API_KEY}. Keep the secret in an environment or a .gitignore'd *.local.env.json.
  • Basic — the Auth tab's Basic mode writes curl's --user "${USER}:${PASSWORD}", so the credentials live in variables rather than a hand-encoded header.

OAuth2

On the Auth tab, choose OAuth2 and pick a grant. The config (grant, token URL, client id, scope, …) is saved in the request's frontmatter, and can also be set once on a collection.json / folder.json so every request underneath inherits it. When you acquire a token it's stored in a session variable (default ${OAUTH_TOKEN}) — reference that from your request's own Authorization header.

client_credentials & password

These are non-interactive: click Acquire token and Freepost POSTs to the token URL (HTTP Basic client auth for confidential clients; the password grant also sends the username/password) and stores the returned access token. Because there's no browser step, these work headlessly in the CLI too.

authorization_code (interactive sign-in)

This is the browser-based flow used by most modern providers (Google, Okta, Auth0, …). Fill in the Authorization URL, token URL, and client id; the Redirect URI is optional — leave it blank and Freepost picks a loopback port automatically. Then click Sign in:

  1. Freepost starts a temporary loopback listener on 127.0.0.1 and opens the authorize URL in your system browser (never an in-app window — consistent with the zero-network, no-embedded-browser design). PKCE (S256) and a CSRF state are always used.
  2. You log in with the provider. It redirects back to the loopback URI with a code.
  3. Freepost validates state, exchanges the code for a token, and shuts the listener down. You can close the browser tab.
Token cache & refresh
The acquired token is cached under the collection's .freepost/oauth-tokens/ folder (owner-only permissions, and .freepost/ is always git-ignored). On later sends, a still-valid token is reused silently; an expired one with a refresh token is refreshed automatically — no need to click Sign in again until the refresh itself fails.
authorization_code in CI
Headless freepost run can't open a browser, so it can only reuse a token a prior interactive sign-in cached (refreshing it if needed). Sign in once in the desktop app so the token exists under .freepost/, and make sure that cache travels to your CI runner (it's git-ignored by default, so provide it out of band). If no usable token is cached, the run fails with a clear "sign in via the Auth tab" message rather than hanging.

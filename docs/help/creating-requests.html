Creating requests

Every request is a file in your collection: a .curl file for REST and GraphQL, a .ws file for WebSocket, .grpc for gRPC, .mqtt for MQTT, and .mcp for an MCP server call. There's no separate "new GraphQL request" — GraphQL is a body mode on an ordinary .curl request.

Open a collection

Click Open Collection in the sidebar and pick any folder. Any folder containing .curl, .ws, or .workflow.json files works — there's nothing to initialize first. The sidebar tree mirrors that folder exactly, one to one.

Create a new request

Hover a folder in the sidebar and click the + button that appears next to it. A menu offers three kinds of new item:

The + menu on any folder row. Requests and workflows can live anywhere in the tree.

  • New Request (.curl) — a REST request. This is also the starting point for GraphQL (see below).
  • New WebSocket (.ws) — a WebSocket connection.
  • New Workflow — an ordered sequence of existing requests; see the Workflows guide.

REST requests

A new request opens with a method dropdown and URL bar at the top, and a row of section tabs below it: Headers, Body, Auth, Scripts, and Meta.

The Meta tab holds the request's own variable defaults, an optional CA certificate, and a "skip HTTPS validation" checkbox for self-signed backends.

In the Body tab, pick a mode: Raw · JSON (or any raw text/XML), Multipart form (file uploads, text fields), or GraphQL.

Switch to Raw view any time
Every editor has a Form / Raw toggle. Raw shows the exact .curl file as it will be written to disk — useful for copy-pasting into a terminal, or hand-editing something the form doesn't expose yet.

GraphQL requests

Create a normal REST request pointed at your GraphQL endpoint, then switch its Body tab to GraphQL mode. You get a query editor, a typed variables table, and an Introspect schema button that fetches the schema and opens a schema explorer (Query / Mutation / Subscription root types) you can drill into while writing the query.

If a query is a subscription, the tab switches into subscription mode: set a Subscription endpoint (defaults to the request URL), pick a transport — WebSocket or SSE — and click Subscribe. Incoming messages stream live into a Subscription stream log until you click Stop.

Why store GraphQL in a curl file?
The GraphQL query is kept as readable, unescaped text in the frontmatter (graphql.query) — the app generates the JSON-escaped --data in the command from it on save, so the file stays both diffable and directly runnable with bash.

WebSocket requests

A WebSocket request is stored as an annotated websocat command (.ws file). Set the URL, connect, and send messages from the log — or save frequently-used payloads as named message presets so you can fire them with one click during a session.

gRPC requests

A gRPC request is stored as an annotated grpcurl command (.grpc file). Point it at a server address (host:50051), name the method as package.Service/Method, and give it the schema by listing one or more .proto files (with import paths if they reference each other). The request message is JSON, and ${VAR} works in the target, message, and metadata just like everywhere else.

  • Unary calls use Send — the response JSON and the gRPC status code come back in the response panel, and test scripts can assert on them.
  • Server-streaming methods use Stream — each message appends to a live log until the server ends the stream (or you press Stop).
  • plaintext connects without TLS (typical for local dev); insecure TLS uses TLS but skips certificate verification.

Schema comes from local .proto files (server reflection isn't used), so the files travel with your collection and everything stays offline. Client-streaming and bidirectional methods aren't supported yet.

MQTT requests

An MQTT request is stored as an annotated mosquitto_pub / mosquitto_sub command (.mqtt file). The mode toggle picks between the two:

  • Publish — set the broker host/port, topic, QoS, optional retain flag, and a message payload, then Publish. This is one-shot, so it also runs in freepost run.
  • Subscribe — set a topic (wildcards + and # allowed) and Subscribe; matching messages append to a live log until you disconnect. Subscriptions are long-lived, so the CLI skips them.

Username/password and a client id are optional; pointing --cafile at a CA certificate switches the connection to TLS. As always, ${VAR} works in the host, topic, and message.

MCP requests

An MCP request is stored as an annotated MCP Inspector CLI command (.mcp file). Pick a transport — Streamable HTTP for a server already listening on a URL, or stdio for a server program Freepost spawns for you — then a method (tools/call, resources/read, prompts/get, and their list counterparts) and send it. Connect opens a live session that lists everything the server exposes, so you can click a tool to fill the request in rather than typing its schema from memory.

The result comes back like any other response, so test scripts can assert on it — and a stdio server never runs until you've seen the exact command and approved it. See the MCP guide for the full story, including schema snapshots and drift detection.

stdio spawns a program
Switching the transport to stdio replaces the URL with a server command (plus its arguments and environment). Freepost shows you that exact command and asks for approval before it runs — see MCP servers.

Sending & the response

Click Send (or the keyboard shortcut shown on hover). The response panel shows status code and reason, timing, size, headers, and a pretty-printed body. For GraphQL subscriptions, WebSocket connections, gRPC server-streaming, and MQTT subscriptions, the equivalent panel is a live, auto-scrolling message log instead of a single response.

Everything you type is a real curl file
There's no hidden state. Whatever you build in the form is exactly what gets pretty-printed to disk on save — headers, body, auth, and scripts included. See Saving & organizing for how saves and variables work.

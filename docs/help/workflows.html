Workflows

A workflow plays a set of requests one at a time, strictly in order — each step waits for the previous one to fully finish (response received, test script run) before the next fires. It's stored as a .workflow.json file, and can live anywhere in your collection tree.

Creating a workflow

Use the + menu on any folder and pick New Workflow (see Creating requests). The workflow editor has a description field, a data file field, and a list of steps — each step is just a collection-relative path to an existing request.

Type a request's path (or use the field below the list) and press Enter to add a step. Check expect error on a step that's supposed to fail.

Running a workflow

Click ▶ Run. Each step shows a live status icon as it executes:

IconStatusMeaning
✓passedSucceeded normally.
⚠expected-errorMarked expect error, and it did error — as intended.
⚠unexpected-successMarked expect error, but it succeeded — flagged as a warning; the run continues.
✗failedErrored without expect error set — the workflow halts here.
○skippedNever reached because an earlier step halted the run.

A step counts as an error if the request fails at the transport level, returns HTTP 4xx/5xx, or its test script has a failing assertion.

Passing data between steps

Workflows never override a request's own variables — every step resolves exactly as it would standalone (see variable resolution). Instead, requests write to the session as they run: capture a token in a pre-request or test script on step 1, and step 3 reads it from the session automatically. Just remember to leave that variable blank on step 3's Meta tab so the session value isn't shadowed.

Data-driven runs

Set a Data file (a CSV or JSON file, ideally kept inside the collection) and the workflow runs once per row — each row's columns are loaded into the session at the start of that iteration, so a request can reference ${email} or ${user_id} straight from the data file. After the run, each iteration gets its own pass/fail summary.

Broken references & auto-heal

Steps reference requests by collection-relative path, so external deletes/renames/moves can break a workflow. Freepost validates every step reference on open and on every filesystem change while the workflow is open — broken steps are flagged with a relink (pick a replacement) or remove step action, and a workflow with broken references can't be run until they're resolved. Renames and moves done inside the app auto-heal every workflow that references the moved file, so references only break via edits made outside Freepost.

Running headlessly
Workflows also run from the command line for CI: freepost run <collection> --workflow <path> uses the same engine as the app.

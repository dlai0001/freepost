Saving & organizing

There's no database and no sync. A collection is a folder; saving a request writes a file into it. Everything below is just filesystem structure plus a small, predictable set of variable scopes.

Saving

Save (button, or the shortcut shown on hover) rewrites the request to its canonical form: long-form flags, one per line, normalized quoting — the same "gofmt for curl" treatment every time, so diffs stay small and predictable. Two things are always preserved even though the file is rewritten:

  • Frontmatter keys the app doesn't know about are kept verbatim.
  • Standalone # comment lines you wrote in the body stay in place.

If a file doesn't parse as valid Freepost shell (pipes, conditionals, multiple commands), it's shown in the tree marked invalid, stays editable as raw text, and is never rewritten by the app — a parse error can't corrupt hand-written content.

Collections & folders

A collection is whatever folder you opened with Open Collection. Sub-folders in the sidebar are ordinary folders on disk. Rename or move a request or folder in the app and the file/folder on disk is renamed or moved to match — and the reverse holds too: the app watches the filesystem, so edits made outside it (a git pull, an editor, a script) show up immediately.

A collection.json (collection root) or folder.json (any subfolder) sidecar file can set defaults that every request under it inherits: default headers, auth, a CA certificate, proxy settings, and pre-request/test scripts that wrap every request in that folder.

Labels & search

Add labels to a request in its Meta tab (stored as frontmatter label:). The sidebar search box matches free text against name, description, URL, and method, and supports a label: filter — click a label chip on any result to filter by it instantly.

Environments

An environment is a saved set of variables stored as a *.env.json file under an environments/ folder. Pick one from the dropdown at the bottom of the sidebar, or manage them (add, rename, edit values) from the gear icon next to it — a two-pane manager listing every variable across your environments side by side. Secrets left blank in *.env.json and instead put in a git-ignored *.local.env.json stay out of version control.

Session

Click Session in the sidebar to open the session panel: an in-memory table of runtime variables, inspectable and clearable. Pre-request and test scripts write here (pm.variables.set(...) and friends — see the scripting guide). It's cleared when the app restarts and never touches disk.

How variables resolve

Every ${VAR} reference resolves against three tiers, from strongest to weakest:

TierWhere it livesWins when…
1. Meta (request parameters) The request's own ${VAR:-default} assignment block, edited on its Meta tab. Its value is non-empty — this is the strongest tier and overrides session and environment for that one request.
2. Session The runtime store — see above. The Meta value is blank, and something (a script, a workflow, a data-driven run) has written to the session.
3. Environment The selected *.env.json. Nothing stronger set a value.
A non-empty Meta value always wins
If a variable is meant to be captured from a script or supplied per-row by a data-driven workflow run, leave it blank (or required with ${VAR:?}) on the Meta tab. A literal Meta value shadows the session value of the same name every time.

History & saved examples

Every send is recorded to request history (open it from the top bar), kept under a git-ignored .freepost/ folder inside the collection with a default retention of the last 500 entries. Separately, you can save a response as a curated example — committed alongside the request as a <Request name>.examples.json sidecar, useful as living documentation that travels with the request through renames and moves.

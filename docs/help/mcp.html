MCP servers

Freepost speaks the Model Context Protocol as a first-class request kind. Point it at an MCP server over stdio or Streamable HTTP, browse the tools, resources, and prompts it exposes, call them — and then do the thing an MCP inspector can't: assert on the result with pm.* tests and run those assertions headlessly in CI. A schema snapshot turns silent tool-signature drift into a failing build.

The .mcp file

An MCP request is saved as a .mcp file, and — like every other request kind in Freepost — it is a real, runnable command: an MCP Inspector CLI invocation, with the usual YAML-in-comments frontmatter on top. bash runs it; the app executes the same call through the official MCP SDK. 

#!/usr/bin/env bash
# ---
# description: Add two numbers via an MCP server
# label: [mcp]
# scripts:
#   test: |-
#     pm.test("sums 20 + 22", () => {
#       pm.expect(pm.response.json().content[0].text).to.contain("42");
#     });
#     pm.test("is not a tool error", () => {
#       pm.expect(pm.response.json().isError).to.not.equal(true);
#     });
# ---

MCP_URL="${MCP_URL:-http://localhost:3011/mcp}"

npx @modelcontextprotocol/inspector \
  --cli \
  "${MCP_URL}" \
  --transport http \
  --method 'tools/call' \
  --tool-name 'get-sum' \
  --tool-arg 'a=20' \
  --tool-arg 'b=22'

Create one with New MCP from the sidebar's right-click menu. ${VAR} expansion works everywhere it does elsewhere — in the URL, headers, tool arguments, the server command, and its environment — so the same request runs against local and staging servers by switching environment.

Two transports

TransportWhat it is
Streamable HTTP A server already listening on a URL. Add Authorization and other headers in the Headers box.
stdio A server program on your machine that Freepost spawns as a subprocess and talks to over its stdin/stdout. You give the command (e.g. npx), its arguments, and any environment variables it needs.
stdio asks before it runs anything
A .curl file is inert data. A stdio .mcp file names a program to run — so opening a collection somebody shared with you must never silently execute code. The first time a given server command is spawned in a given collection, Freepost shows you the exact command and asks for approval; the answer is remembered per collection + command, and it is stored in the app's own data directory, never inside the collection (a collection author could otherwise ship their own approval). The CLI doesn't prompt — running freepost run on a collection is itself your authorization — so use --no-mcp-spawn to skip stdio requests in CI.

The MCP tab

Pick a transport and a method, fill in the target, and hit Send. Six methods are supported:

MethodNeeds
tools/list—
tools/callA tool name and tool arguments (key=value, one per line)
resources/list—
resources/readA resource URI, e.g. demo://greeting
prompts/list—
prompts/getA prompt name and prompt arguments

Connect opens a live session instead of a one-shot call. Freepost introspects the server and lists its tools, resources, and prompts — click any one of them to fill the request in with its name and parameters, so you don't have to type a schema you can already see. Server notifications (including progress) stream into the session log while you're connected. Disconnect closes it.

Tool arguments are typed against the tool's own inputSchema before the call goes out: a parameter declared number is sent as 20, one declared string stays "20". You write key=value; the schema decides what that value is.

Two failure axes, kept apart

MCP can fail in two very different ways, and collapsing them makes every assertion subtly wrong. Freepost maps them onto distinct statuses so you can assert on either:

StatusMeaning
200 OKThe call succeeded.
500 TOOL_ERRORThe tool ran and reported failure — the result carries isError: true. Its content is still there to assert on. (An unknown tool name lands here too, not in PROTOCOL_ERROR.)
502 PROTOCOL_ERRORThe call never got that far: the transport failed, the subprocess wouldn't spawn, the server rejected the request.

The response body is the raw MCP result, so pm.response.json() gives you content, structuredContent, and isError directly: 

pm.test("the tool did not error", () => {
  pm.expect(pm.response.json().isError).to.not.equal(true);
});
pm.test("structured output has a total", () => {
  pm.expect(pm.response.json().structuredContent.total).to.equal(42);
});

Schema snapshots & drift detection

The loudest complaint in the MCP ecosystem is silent schema drift: a server renames a tool or retypes a parameter between versions, and everything downstream breaks with no error anywhere. Freepost's answer is its whole thesis applied to MCP — write the server's introspection surface to a file in your collection, review it in git diff, and fail the build when it changes.

Snapshot schema records the server's tools, resources, and prompts as <request>.mcp.snapshot.json, next to the request. Commit it. Check drift re-introspects the live server and diffs it against that file:

ChangeVerdict
A tool, resource, or prompt disappearsBreaking — fails CI
A tool parameter disappears, changes type, or becomes newly requiredBreaking — fails CI
Anything new: a tool, a resource, a prompt, an optional parameterAdditive — warns only

The same two actions exist on the command line, so this runs in CI:

freepost mcp snapshot ./collection   # record each server's schema
freepost mcp check ./collection      # exit 1 on breaking drift

When the server calls back

Some servers ask the client for something mid-call. Freepost never calls an LLM and never blocks a headless run waiting for a human, so both callbacks are answered from frontmatter — scripted, deterministic, reviewable: 

# ---
# mcp:
#   sampling:
#     text: "a canned model completion"
#   elicitation:
#     action: accept        # accept | decline | cancel
#     content:
#       city: "Portland"
# ---

Without an mcp.sampling block, a server that asks for an LLM completion gets a protocol error rather than a silent stub. Without an mcp.elicitation block, an elicitation request is declined.

Callbacks need a session
Sampling and elicitation are server-to-client calls, so they only work where a persistent session exists: stdio, or a session-ful HTTP server. A stateless Streamable HTTP server that builds a fresh instance per POST never sees the handshake and can't call back. Progress notifications work either way — they stay within a single request.

Try it offline

The repo ships mock servers and a wired-up collection for every protocol, MCP included, so you can exercise all of this without touching the internet: 

npm run fixtures                 # starts MCP over http (:3011) and stdio, plus the rest
freepost run fixtures/collection

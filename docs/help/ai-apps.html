Use Freepost from an AI app

MCP servers covers Freepost calling out to an MCP server. This is the reverse: Freepost is an MCP server, so Claude Desktop — or any MCP client — can open your collection and work in it. Hand it an OpenAPI spec or a GraphQL schema and ask for tests; it writes the request files, runs them, reads the assertion results, and fixes what fails. Everything it does is a file on your disk and a git diff you review.

What you can ask for

Once connected, prompts like these do the whole job end to end:

  • "Create test cases for the endpoints in this Swagger schema: ./openapi.json"
  • "Cover every query and mutation in the GraphQL schema at http://localhost:4000/graphql"
  • "Run the Users folder and fix any test that fails."
  • "Add an assertion to every request that the response time is under 500ms."

The app is watching the folder, so files the AI writes appear in your sidebar as they land.

Connect: from the app

Tools ▸ MCP Server starts a listener for the collection you currently have open. The menu shows the URL it's listening on — http://127.0.0.1:7599/mcp by default — and Copy AI app config snippet puts the config on your clipboard: 

{
  "mcpServers": {
    "freepost": {
      "type": "http",
      "url": "http://127.0.0.1:7599/mcp"
    }
  }
}

It is off by default, and stays off until you ask for it — every session. That's deliberate: an always-on server means every AI conversation you have carries Freepost's tools whether or not that conversation is about your API. Turn it on for the work, turn it off after.

If you open a different collection while the server is running, it stops. A server started for one collection will not quietly begin serving another. To change the port, set "mcpServerPort" in settings.json in the app's user-data folder.

Connect: from the command line

No app needed — the AI app launches Freepost itself, which is what Claude Desktop expects. Point it at the collection folder: 

{
  "mcpServers": {
    "freepost": {
      "command": "freepost",
      "args": ["mcp", "serve", "/path/to/your/collection"]
    }
  }
}

Or run it directly to see what it exposes:

freepost mcp serve ./my-collection

The tools

ToolWhat it does
get_format_specThe request-file grammar and a starter to copy, per kind. The AI reads this before its first write.
list_collectionThe request/folder/environment tree.
read_requestA request file's text, and whether it parses.
write_requestCreate or update a request — including its pm.* test script. Parsed first; a bad file is rejected, not written.
move_pathRename or move a request or folder.
delete_pathDelete a request or folder.
run_requestExecute a request for real and return the response plus its test results.
import_openapiGenerate one request per operation from an OpenAPI/Swagger spec.
read_environment / write_environmentList, read, and write non-secret environments.
describe_graphql_schemaSummarize a GraphQL schema — from a live endpoint, an SDL file, or pasted SDL.

How it writes a request

The AI sends the whole file text, exactly as you'd write it by hand — there's no parallel JSON API that could drift from the format. Freepost parses it first: a file that doesn't fit the grammar is rejected with the line number and nothing is written, so the model can correct itself. A file that does fit is re-serialized to canonical form before it lands, which means AI-written files are formatted identically to app-written ones and git diff stays readable.

GraphQL

A GraphQL request is a .curl file with a graphql: frontmatter block — there's no separate extension. The AI starts with describe_graphql_schema, which returns the queries and mutations with their argument types (not the whole schema, which would swamp its context), then writes one request per operation: 

#!/usr/bin/env bash
# ---
# description: List users
# graphql:
#   query: |-
#     query Users {
#       users {
#         id
#         name
#       }
#     }
#   variableDefs: []
# scripts:
#   test: |-
#     pm.test("status is 200", () => pm.response.to.have.status(200));
#     pm.test("no graphql errors", () => {
#       pm.expect(pm.response.json().errors).to.equal(undefined);
#     });
# ---

GQL_URL="${GQL_URL:-http://localhost:4000/graphql}"

curl --request POST \
  --url "${GQL_URL}" \
  --header 'Content-Type: application/json' \
  --data '{"query":"query Users {\n  users {\n    id\n    name\n  }\n}"}'

Note the second assertion: a GraphQL error still comes back as HTTP 200, so asserting on status alone would pass a broken query. Subscriptions are long-lived and can't be run one-shot — run_request says so rather than hanging.

What it can't do

An AI client is driven by a model that may be reading things you didn't write — a spec someone sent you, a web page. "The AI asked" is not authorisation, so the boundaries are enforced, not advisory:

  • It never leaves the collection. Every path is collection-relative. Traversal (../), absolute paths, .git/, and .freepost/ (your secrets, cached tokens, and request history) are refused.
  • It never reads your secrets. *.local.env.json — the git-ignored file where secrets belong — is neither listed nor readable. A variable marked secret: true never gets a literal default written to disk. The AI can use a secret by running a request that resolves it; it can't read one.
  • It never silently spawns a program. A stdio .mcp request names an executable. Over the app's HTTP server, the AI can only run servers you already approved by hand. From the CLI, typing mcp serve is the authorisation — the same doctrine as freepost run — and --no-mcp-spawn opts out.
  • Responses are truncated. A large response body comes back clipped, so one chatty endpoint can't blow out the model's context.

One tradeoff to know about: the app's HTTP listener binds to 127.0.0.1 only, but carries no auth token. While it's on, any program running as you on this machine can reach it. That's the price of it being one menu click — which is also why it's off by default. The CLI's stdio transport has no socket at all, so it doesn't have this property.

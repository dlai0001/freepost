Freepost help

Freepost is an offline API client: REST, GraphQL, WebSocket, gRPC, MQTT, and MCP requests, saved as pretty-printed curl / websocat / grpcurl / mosquitto / MCP Inspector commands in a folder you choose. No account, no cloud, nothing leaves your machine except the requests you send. These guides cover the everyday workflow — creating requests, calling MCP servers, importing existing collections, recording real traffic through a local proxy, organizing what you build, chaining requests into workflows, and writing pre-request / test scripts.

Concepts, quickly

Freepost reuses ordinary filesystem concepts instead of inventing new ones. If you know what these five terms mean, you know the app:

Collection
Any folder you open with Open Collection. There's no separate "workspace" — the folder is the whole thing, and it's portable to any machine.
Request
A single file: .curl (REST or GraphQL), .ws (WebSocket), .grpc, .mqtt, or .mcp (an MCP server call). The filename is the request's name; the file is valid, runnable shell.
Folder
A subfolder in your collection. The sidebar tree mirrors the folder tree on disk exactly — rename or move either side and the other follows.
Workflow
A .workflow.json file: an ordered list of requests played back sequentially, one at a time.
Environment / Session
An environment is a saved set of variables (a *.env.json file) you pick from the sidebar. The session is the app's in-memory scratch space for the current run — scripts write to it, and it's never saved to disk.
Why curl?
Every request is stored as a real, runnable shell script with a YAML frontmatter comment block for metadata (description, labels, scripts, variables). bash the-file.curl runs it outside the app. git diff reviews it. That's the whole storage model — see the format specification for the full grammar.
